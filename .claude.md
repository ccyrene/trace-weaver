# Claude Handoff: TraceWeaver

You are continuing development on **TraceWeaver**, a Python CLI tool that statically scans Apache Airflow DAG repositories and extracts lineage candidates.

## Product intent

TraceWeaver should be a separate service/tool from DAG code. It takes a DAG repository as input and produces structured data lineage output as CSV, JSON, and later Postgres/OpenLineage-compatible records.

Core questions TraceWeaver must answer:

1. What data source is used and where does it output?
2. Which DAG and task are responsible?
3. Which Python functions are involved?

## Current MVP scope

The zip contains a working stdlib-based CLI:

```bash
traceweaver scan --repo-path examples/sample_dags --output csv --output-dir outputs/csv
```

The MVP intentionally avoids importing Airflow. It uses Python AST parsing only, so sample DAGs can be scanned even when Airflow is not installed.

## Current modules

- `traceweaver.cli` - argparse CLI entrypoint.
- `traceweaver.scanners.repo_scanner` - walks Python files and orchestrates scanning.
- `traceweaver.scanners.python_ast_scanner` - extracts DAGs, tasks, function calls, and SQL strings.
- `traceweaver.scanners.sql_scanner` - regex-based SQL input/output dataset extraction.
- `traceweaver.models` - dataclass models for jobs, datasets, edges, function calls, and scan results.
- `traceweaver.exporters.csv_exporter` - writes CSV artifacts.
- `traceweaver.exporters.json_exporter` - writes raw JSON artifact.

## Development priorities

### Priority 1: Improve Airflow extraction

Support more DAG patterns:

- `with DAG(dag_id="...") as dag:`
- `dag = DAG(...)`
- `@dag(...)` decorated factory functions
- `@task` decorated functions inside DAG factories
- `PythonOperator`, `BashOperator`, `PostgresOperator`, `BigQueryInsertJobOperator`, `SnowflakeOperator`, `KubernetesPodOperator`
- dependency extraction via `>>`, `<<`, `set_upstream`, `set_downstream`

### Priority 2: Better SQL lineage

Replace or augment regex SQL scanner with `sqlglot`:

- Extract source tables from `FROM` and `JOIN`.
- Extract target tables from `INSERT INTO`, `CREATE TABLE AS`, `CREATE OR REPLACE TABLE`, `MERGE INTO`.
- Preserve SQL dialect when known.
- Add confidence scoring.

Expected model:

```text
dataset_source --used_by--> dag.task --produces--> dataset_output
```

### Priority 3: Dataset extraction from Python code

Detect common dataset strings:

- S3: `s3://bucket/path`
- GCS: `gs://bucket/path`
- local files: `.csv`, `.parquet`, `.json`
- DB tables from strings: `schema.table`
- Airflow connection IDs from `conn_id`, `postgres_conn_id`, `gcp_conn_id`, etc.

### Priority 4: Postgres exporter

Add optional dependency and exporter:

```bash
traceweaver scan \
  --repo-path ./dags \
  --output db \
  --database-url postgresql://lineage:lineage@localhost:5432/lineage
```

Target tables:

- `lineage_jobs`
- `lineage_datasets`
- `lineage_edges`
- `function_calls`
- `raw_scan_results`

### Priority 5: Runtime roadmap

Do not start with eBPF. The recommended runtime path is:

1. OpenLineage provider in Airflow.
2. TraceWeaver collector API.
3. Python runtime tracer for function-level truth.
4. Optional eBPF agent only as system-level evidence.

## Coding constraints

- Keep the CLI runnable without Airflow installed.
- Never import DAG files directly in the scanner by default.
- Prefer AST/CST parsing over execution.
- Keep output models stable and serializable.
- Add tests for every newly supported DAG pattern.
- Treat extracted lineage as candidates unless verified by runtime events.

## Suggested next tasks

1. Add dependency extraction between tasks.
2. Add `sqlglot` as optional dependency and build `SqlGlotLineageExtractor`.
3. Add Git clone support using GitPython or subprocess git.
4. Add Postgres exporter.
5. Add confidence rules module.
6. Add docs with examples and limitations.

## How to test

```bash
python -m unittest discover -s tests
python -m traceweaver scan --repo-path examples/sample_dags --output csv --output-dir outputs/csv
python -m traceweaver scan --repo-path examples/sample_dags --output json --output-dir outputs/json
```

## Important product decision

TraceWeaver is **Python-first** because Airflow DAGs are Python. Rust can be added later for high-performance repo walking, runtime observers, or eBPF-related components.
